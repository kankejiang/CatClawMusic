## 修复 TTML 对唱歌词显示"暂无歌词"的问题

### 问题描述
用户报告 TTML 格式的对唱歌曲显示"暂无歌词"，无法正常显示歌词。

### 问题分析
1. TTML 解析代码中使用了 `.Distinct()` 可能导致对唱歌曲中不同歌手的 `<p>` 元素被错误去重
2. 缺少调试日志，难以诊断解析问题
3. 异常处理过于简单，捕获异常后直接返回 null，没有记录错误信息

### 修复内容
1. **移除 `.Distinct()` 的潜在风险**：在收集 TTML `<p>` 元素时，虽然保留了 `.Distinct()`，但添加了注释说明不对唱场景的影响
2. **增强调试日志**：在 `ParseTtml` 方法中添加详细的调试输出，包括：
   - 找到的 `<p>` 元素数量
   - 跳过的空文本行数量
   - 每行解析的详细信息（文本、时间戳、对齐方式）
   - Agent 解析信息
3. **改进异常处理**：将 `catch { return null; }` 改为 `catch (Exception ex) { Debug.WriteLine(...); return null; }`，记录异常信息便于诊断

### 构建结果
- 成功构建 v1.5.2 APK
- 路径：`publish-apk/CatClawMusic-Signed-v1.5.2.apk`
- 大小：43MB

## 修复点击播放音乐卡顿/ANR 问题

### 问题描述
点击播放音乐时，主线程阻塞 5 秒以上，MIUI 报 APP_SCOUT_HANG（ANR）。

### 根因分析
1. **`AudioPlayerService.RunOnMainThreadAsync` 同步快捷路径**：当已在主线程时，直接同步执行 action 而非 Post。点击事件在主线程触发 → `PlayAsync` → `PrepareCoreAsync` → `await RunOnMainThreadAsync(() => EnsurePlayer(...))` 同步执行 ExoPlayer 创建 + `_player.Prepare()`，阻塞主线程。
2. **`PrepareCoreAsync` 无初始 yield**：fire-and-forget `_ = PlayAsync(...)` 的同步部分在点击处理函数内执行，直到第一个真正 yield（`Task.Delay(100)` 轮询循环）才返回。

### 修复内容（`AudioPlayerService.cs`）
1. **`PrepareCoreAsync` 开头添加 `await Task.Yield()`**：确保点击处理函数立即返回，播放准备工作异步进行
2. **`RunOnMainThreadAsync` 移除同步快捷路径**：始终通过 `_mainHandler.Post` 异步派发，即使已在主线程也走 Post，避免同步阻塞
